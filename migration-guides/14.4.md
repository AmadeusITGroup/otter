# Minor 14.4 Migration guide

> [!IMPORTANT]
> Please make sure to first follow the ["Upgrade new version guide"](https://github.com/AmadeusITGroup/otter/blob/main/docs/core/UPGRADE_NEW_VERSION.md) before going through these steps.

## @o3r/components â€” `@o3r/transloco` build error

Starting from v14.4, `@o3r/components` dynamically imports `@o3r/transloco` at runtime (with `@o3r/localization` as a fallback).
Both packages are optional peer dependencies, but webpack and the Angular esbuild builder statically resolve all `import()` expressions at build time.
If `@o3r/transloco` is not installed in your project you may encounter the following error during your build **or your tests**:

```
Module not found: Error: Can't resolve '@o3r/transloco' in '.../node_modules/@o3r/components/fesm2022'
```

> [!NOTE]
> If you **do** use `@o3r/transloco` in your application (i.e. it is listed in your `package.json` dependencies), no action is required â€” the package is bundled as usual for both build and tests.

The recommended fix is to use a builder that supports the `externalDependencies` option, so the `ng update @o3r/components` migration can mark `@o3r/transloco` as external automatically for application projects that do not have it installed.
If migrating the builder is not an option, install `@o3r/transloco` as a dev dependency instead â€” this works for **every** builder (including the webpack-based ones):

```shell
npm install --save-dev @o3r/transloco
```

The dynamic `import()` then resolves and the package is bundled normally. If you already added the `IgnorePlugin` to a production webpack config (see below), `@o3r/transloco` still stays excluded from that production bundle â€” installing it only fixes the build/test resolution.

### Build

The recommended builder is the esbuild-based `@angular/build:application`, which supports `externalDependencies`. The migration adds the option automatically:

```json
{
  "projects": {
    "my-app": {
      "architect": {
        "build": {
          "options": {
            "externalDependencies": ["@o3r/transloco"]
          }
        }
      }
    }
  }
}
```

> [!NOTE]
> The `externalDependencies` option only exists on builders that support it (`@angular/build:application`, `@angular/build:karma`, `@angular-devkit/build-angular:browser-esbuild`, `@angular-devkit/build-angular:server`). Webpack-based builders (e.g. `@angular-devkit/build-angular:browser`) reject this option, so the migration skips them and logs a warning.

If your build target uses an `@angular-builders/custom-webpack:*` builder, the `externalDependencies` option is not available â€” the migration cannot update it automatically and will instead log a warning pointing to your webpack config file.
Add the following to the webpack config referenced by `customWebpackConfig.path` so the bundler skips the optional `@o3r/transloco` import:

```js
const { IgnorePlugin } = require('webpack');

config.plugins.push(new IgnorePlugin({
  resourceRegExp: /^@o3r\/transloco$/
}));
```

### Testing

For unit tests, the recommended fix is to migrate the test target from the webpack-based `@angular-devkit/build-angular:karma` to the esbuild-based `@angular/build:karma` builder, which supports `externalDependencies` (the migration then adds the option automatically, just like for the build target):

```json
{
  "projects": {
    "my-app": {
      "architect": {
        "test": {
          "builder": "@angular/build:karma",
          "options": {
            "externalDependencies": ["@o3r/transloco"]
          }
        }
      }
    }
  }
}
```

> [!NOTE]
> The standard `@angular-devkit/build-angular:karma` test builder bundles spec files with its own internal webpack and exposes neither `externalDependencies` nor a webpack config hook. If you cannot migrate to `@angular/build:karma`, install `@o3r/transloco` as a dev dependency so it is bundled normally.

End-to-end tests run against the built and served application, so they require no specific change for `@o3r/transloco` â€” once your build is fixed (e.g. Playwright e2e), they work as before.

## @o3r/components â€” remove `@o3r/localization` when using `@o3r/transloco`

`@o3r/components` now resolves the translation service by trying `@o3r/transloco` first and falling back to `@o3r/localization`.
If your application uses `@o3r/transloco`, you no longer need `@o3r/localization` as a dependency for placeholder translation to work and can remove it from your `package.json`.
